Anvil-MC — User Guide

Version 7.0.2 · Minecraft Bedrock Edition addon combiner

What is Anvil-MC?

Anvil-MC takes several Bedrock behavior and resource packs and stitches them into a single, ready-to-use pair of output packs. Instead of merging everything by hand, it handles JSON combination, flags conflicts for you to resolve, joins scripts together, and applies compatibility repairs automatically.

Quick Start

  1. Load your packs — Open the Merger tab and use Add Files to pick .mcpack, .mcaddon, or .zip archives. Add Folder lets you point at an already-unpacked pack.
  2. Inspect first — Hit Check Packs to preview how the selected packs will be grouped by their @minecraft/server API version and to see achievement compatibility before you commit.
  3. Pick a destination — Set the Output Directory where the finished packs should land.
  4. Run it — Press Start Merging. The progress bar tracks each stage, and Cancel is available if you need to abort.

How a Merge Works

Behind the scenes, a merge runs through these stages:

  1. Intake — Every pack is opened and checked for a valid manifest and sane folder layout.
  2. Version snapshot — The highest format versions seen across all inputs are recorded.
  3. Achievement check — Whether the result can stay achievement-friendly is evaluated up front.
  4. Combine — Files are sorted into behavior/resource buckets, then merged per type:
    • JSON is united with conflict awareness.
    • Scripts are concatenated with duplicates stripped.
    • .lang files are joined key by key.
    • Binary assets (textures, audio) follow a first-in wins rule.
    • Entities, items, and blocks are matched by identifier.
  5. Manifest — One coherent manifest is written for each output pack.
  6. Auto Fixers — Addon-specific repairs are applied (optional).
  7. Packaging — Results are zipped and the workspace is cleaned up.

Resolving Identifier Clashes

If two sources define the same identifier (an entity, item, block, and so on), a dialog interrupts the merge so you can decide. Your options:

  • Deep merge — Blend both definitions into one. Best when the two entries are the BP and RP halves of the same addon.
  • Pick a winner — Keep one pack's version and drop the other. Use this when the definitions genuinely conflict.

Script API Version Groups

Addons that target different @minecraft/server versions can't share a single script environment. Anvil-MC detects this and splits the work into one output group per API version. Files that aren't version-sensitive — like player.json and HUD definitions — are shared across every group so nothing gets lost.

Subpack Variants

Some packs ship multiple subpack options (for example, low- and high-resolution texture sets). When Anvil-MC finds these, it asks which variant you want before continuing, so the merged output matches your target devices.

Auto Fixers

Toggle Auto Fixers from the Settings tab. When enabled, the framework repairs frequent addon headaches on the fly:

  • Converting legacy item and block formats
  • Filling in missing item or block definitions
  • Removing empty JSON files
  • Cleaning up placeholder animation controllers
  • Relocating misplaced files to their correct paths
  • Refreshing outdated sound definitions
  • …and other edge cases

MCPacker Tab

A standalone utility for repackaging loose folders. Choose Pack to turn a manifest-bearing folder into a .mcpack, or Unpack to do the reverse. Add the folders you want, set an output location, and press Start.

List Maker Tab

Use this tab to get a quick overview of a pile of pack files. Organize by Date sorts them chronologically and nests each under its Script API version group. Export List writes the resulting tree out to a file you can share or archive.

Settings Tab

Configure global behavior here:

  • Language — Interface language (takes effect after restart).
  • Auto Fixers — Turn the compatibility repair layer on or off.
  • Modpack organization — Keep files grouped by their source pack in the output.
  • Merge by version — Split output by @minecraft/server version instead of forcing a single target.
  • Customize dialog — Pop up the pack customization window after each merge completes.
  • Show linked packs — Display linked pack relationships at the end of a run.
  • Default output — Pre-fill the output directory for every merge.

When Something Goes Wrong

If a merge fails, walk through these steps:

  1. Read the message in the dialog — it usually names the offending pack.
  2. Open error_log.txt in the app data folder for the full traceback.
  3. Confirm every input pack has a readable manifest.json.
  4. Check that the output folder is writable and not full.

Tips for Smooth Merges

  • Run Check Packs before committing — it catches version mismatches early.
  • Prefer Deep merge for the BP/RP halves of one addon.
  • Glance over the merge log afterward for warnings about redeclared variables.
  • Keep your source packs in dedicated folders so re-running a merge is painless.
